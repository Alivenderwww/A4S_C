## 三、官方回复与契约澄清

### 回复 1：GEMM 越界输出（R203 泛化）

**来源**：组委会对 undersized C 的正式回复。
**内容**：对所有 GEMM 数值类型（FP4/FP8/FP16/BF16/FP32/FP64/INT4/INT8/INT32），
合法 shape + 正确大小的 A/B + undersized C 必须返回 `AEC_ERROR_ISA_TRAP`。
**R203 hidden 测试只检查 INT8**，但契约适用于所有 dtype。
**测试覆盖**：见 `tests/test_official_contradictions.py::test_gemm_oob_all_dtypes`：
在合法 A/B shape（目前为 2×3×2）下，将 C 分配为 1 字节后调用各 dtype GEMM，
记录返回值。若返回 ISA_TRAP 则契约满足；若返回其他 error code 或 SUCCESS 则
标记为偏离（需要按设备具体行为单独归因，不预先认定为 Runtime 错误）。

### 回复 2：Vector one-past = INVALID_ARGUMENT（R204）

**来源**：组委会对 R204 hidden 检查的正式回复。
**内容**：one-past-the-end count 对 AXPY/DOT/NRM2 必须返回 `INVALID_ARGUMENT`，
**不是** `ISA_TRAP`。Runtime 应在 Host 侧通过 `allocation_fits` 检查拦截。
**不应改变**为 ISA_TRAP。
**测试覆盖**：见 `tests/test_official_contradictions.py::test_vector_one_past_is_invalid_not_trap`。

### 远端行为观察——已知不确定项

以下测试结果取决于远端设备行为，不能由本地审计预先判定：

1. **`test_gemm_oob_all_dtypes`** 和 **`test_gemm_oob_all_fp8_formats`**：
    如果设备对不同 dtype 的 OOB 行为不一致（某些 dtype 返回 INVALID_ADDRESS
    而非 ISA_TRAP），该 dtype 会 FAIL。测试输出按 dtype 和 api 归因，
    不预先假设是 Runtime 还是 device 错误。

2. **`test_nan_probe_all_gemm`** 和 **`test_nan_probe_all_vector`**：
    如果设备产生非 canonical NaN，测试本身不 FAIL（它只是探测），
    但会在 stdout 打印 `UPSTREAM-MISMATCH`，并通过 `AEC_NAN_REPORT_PATH` 环境变量
    写入纳米报告（默认临时目录，不写源码目录；原子写失败会导致测试 FAIL）。

3. **`test_gemm_m_257_is_invalid`** / **`test_gemm_n_257_is_invalid`** / **`test_gemm_k_257_is_invalid`**：
     预期 INVALID_ARGUMENT，若远端返回 SUCCESS 或其他错误代码则 FAIL；
     shape 校验拦截在 Host Runtime 中实现，结果归因于 Runtime。
