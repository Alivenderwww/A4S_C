## ⭐ 更新（2026-07-14，真实 GPR spill 落地——覆盖本文 P0 断言）

- **P0「spill/reload 钳位桩」已消除**：`linear_scan.cpp` 实现真实窄寄存器（b32）spill——保留 R251（LMEM 地址）+ R252..R255（值）作 scratch，可分配区 R1..R250，压力超 250 时按 SpillAtInterval 驱逐最长活跃区间到 per-thread `.lmem`（`LD.lmem`/`ST.lmem`）。**不再是"超 255 静默钳到 R255 → 算错"**。无溢出时输出与旧扫描逐字节一致。frontier 9/9 PASS（gpr-live + pair-live 六条从 XFAIL 转真实 PASS；pair 因 `narrowAddr` 塌成 32-bit 被同一 spill 覆盖）。本地 sim 21 例（3 档 × 7 count）+ 官方 CModel（`--lmem-size 4096`）均数值正确。
- **P0「predicate>8 分配/溢出」已消除**：新增 `src/regalloc/pred_alloc.cpp`（listSchedule 后、linearScan 前）——真实活跃度重编号虚拟 predicate → P0..P6，保留 P7 为重物化 scratch，活跃超 7 个时溢出到 GPR（定义 `CMPP Pd`→`CMP Rv` 写 0/1 布尔；每个 guard 前插 `CMPP.ne P7, Rv, Rzero` 重物化，引入的 GPR 交给 linear-scan 处理）。distinct predicate ≤8 时空转（公开 T1–T5 不变）。frontier 9/9 PASS 且 `expected_failures.json` 清空。官方 CModel 验证：pred-live-9/12/16 rc=0 正确 + 判别性测试（12 个值各异、跨块活跃、branch-guard 使用、溢出 5 个）CModel sum 正确。**注**：C1 子集只支持谓词化分支 `@%p bra`；谓词化算术 `@%p add` 的 guard 在 ir_builder 被丢弃（公开 T1–T5 只用前者）。
- **旁支发现（更新 f16 P0 定性）**：**AEC ISA 无 16-bit 内存访问**——`LD` 仅 `b32/b64/u32/s32/f32`、`ST` 仅 `b32/u32/s32/f32`（spec §4.1）。故 f16 元素级 store（`ST.b32` 写 4 字节覆盖相邻元素、并行竞态）是 **ISA 层限制**，非编译器可干净修复；且公开 T5 GEMM 存 `f32` 不存 f16，f16 缺口实际风险低于原估。**至此 gpr/pair/predicate 三类压力边界全部消除。**

### 回归已抓出的真实 bug（4 个）

1. **R5 跨循环 clobber**：reuse 用例——旧线性扫描不跨回边延伸区间。

4. **b64 寄存器对 clobber**：寄存器压力变异暴露——allocator 不为 pair 保留 Rd+1。

## 六、Performance Model TODO

### 参数已由 hint.md 公布（无可执行 C-model）

`Track-C/hint.md` 公布了**类 NVIDIA GPU 目标平台参数**（两平台 A100 级 / H100 级），作为本阶段目标硬件指标，要求参赛者据此**自建 Performance Model**。仍无可执行 cycle model（属赛道 B 交付物）。

**⚠️ 评分用的是 Track-B §7 的抽象内存模型,不是 hint.md 的 600**：Track-B §7 明确 GMEM/LMEM latency = **32 cycles**、line = **128 bytes**、1 req/cycle、**16 outstanding**、in-order，并写「该 model 是比赛统一抽象,不代表某种具体 DRAM/HBM。Hidden performance tests 使用相同公开参数」。官方 `total_cycles`（Track-B CModel §8）用此模型 → **我们 sim 的 GMEM=32 已对齐,不要改成 600**。

hint.md 的分级延迟（Shared 20/L1 40/L2 200/**HBM 600**、RF 256KB/SM、L2 40/50MB、PTX 9.3）是"目标硬件指标",供构建 NVIDIA 式性能模型/理解真实瓶颈,**不是评分模型**。

真正的建模缺口是：sim 尚未建模 §7 的 **128-byte line 合并（memory_transactions）** 和 **1 req/cycle + 16 outstanding 吞吐**（只建了 32c 延迟）—— 这才是待补的忠实建模项。

## 七、限制与风险（完整清单）

### 自建 oracle 的边界

1. **不是官方 golden**：sim 语义与真正 golden 若有出入，sim PASS ≠ 官方通过。
2. **单 warp 顺序**：无 atomics/barrier/smem 跨线程建模（公开 kernel 用不到）。
3. **FP 精度**：FMA 用 float64 中间量模拟单次舍入；MAD 两次舍入。
4. **BRX 发散检测**：只验证 uniform；正确实现应无 DIVERGE（已谓词化）。

### 工程风险

1. ✅ ~~identity stub 静默通过~~：const_prop 已实装（真实折叠+传播）；mem_coalesce 未接入但冗余/不变量访存由 CSE/LICM 覆盖，非正确性风险。
2. ✅ ~~spill 钳位可能算错~~：已实真 spill（`ST.lmem`/`LD.lmem`），压力超限正确溢出，不再冲突。
3. ⚠️ **f16 边界**：AEC ISA **无 16 位内存访问**（LD/ST 仅 32/64 位）→ f16 元素级 load/store 是 ISA 层限制，非编译器可修；公开 T5 存 f32 → 实际风险低。
4. **本地性能代理与真实延迟有偏**：est_cycles 已改延迟感知（与 sim 同口径，排序一致），但三套代理的 GMEM 延迟仍是 32，而 hint.md 真实 HBM≈600 → 需按分级延迟重标定；`DEFAULT_CONFIG=O2`，官方 T_default 口径 Q7 待确认。
5. **CVT 已确认**：源类型编码 = Track-B §5.3（dst[6:3]/src[13:10]），已由 `cvtff` 执行 golden 验证。隐藏测试的精度组合覆盖仍受 ISA 类型面约束（无 fp8/fp4/int4 标量）。
6. **多 kernel 未实现**：若隐藏测试有多个 entry 函数，当前只编译第一个。
7. **依赖 `public/` 路径**：`Makefile` 和测试脚本引用 `../public/Track-C/`，需确保路径在各环境一致。
