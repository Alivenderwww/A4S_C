### 回归已抓出的真实 bug（4 个）

1. **R5 跨循环 clobber**：reuse 用例——旧线性扫描不跨回边延伸区间。
2. **CVT 源类型未编码**：encoder 只编 dst type，与 golden 不匹配。
3. **跨块 CSE**：非 SSA 假名导致错误复用。
4. **b64 寄存器对 clobber**：寄存器压力变异暴露——allocator 不为 pair 保留 Rd+1。

## 六、Performance Model TODO

### 参数已由 hint.md 公布（无可执行 C-model）

`Track-C/hint.md` 公布了**类 NVIDIA GPU 目标平台参数**（两平台 A100 级 / H100 级），作为本阶段目标硬件指标，要求参赛者据此**自建 Performance Model**。仍无可执行 cycle model（属赛道 B 交付物）。

**⚠️ 评分用的是 Track-B §7 的抽象内存模型,不是 hint.md 的 600**：Track-B §7 明确 GMEM/LMEM latency = **32 cycles**、line = **128 bytes**、1 req/cycle、**16 outstanding**、in-order，并写「该 model 是比赛统一抽象,不代表某种具体 DRAM/HBM。Hidden performance tests 使用相同公开参数」。官方 `total_cycles`（Track-B CModel §8）用此模型 → **我们 sim 的 GMEM=32 已对齐,不要改成 600**。

hint.md 的分级延迟（Shared 20/L1 40/L2 200/**HBM 600**、RF 256KB/SM、L2 40/50MB、PTX 9.3）是"目标硬件指标",供构建 NVIDIA 式性能模型/理解真实瓶颈,**不是评分模型**。

## 七、限制与风险（完整清单）

### 自建 oracle 的边界

1. **不是官方 golden**：sim 语义与真正 golden 若有出入，sim PASS ≠ 官方通过。
2. **单 warp 顺序**：无 atomics/barrier/smem 跨线程建模（公开 kernel 用不到）。
3. **FP 精度**：FMA 用 float64 中间量模拟单次舍入；MAD 两次舍入。
4. **BRX 发散检测**：只验证 uniform；正确实现应无 DIVERGE（已谓词化）。

### 工程风险

1. **identity stub 静默通过**：const_prop/mem_coalesce 返回 false（无改变），不会报错，但可能错过正确性和性能分。
2. **spill 钳位可能算错**：超寄存器时映射到同一物理寄存器 = 冲突，可能产生错误结果。
3. **f16 边界越界**：`ld.global.u16` 降为原始字节地址 `LD.gmem.b32`（读 4 字节），末元素在 gmem 末尾多读 2 字节；奇数半字地址语义也有风险。
4. **本地性能代理与真实延迟有偏**：est_cycles 已改延迟感知（与 sim 同口径，排序一致），但三套代理的 GMEM 延迟仍是 32，而 hint.md 真实 HBM≈600 → 需按分级延迟重标定；`DEFAULT_CONFIG=O2`，官方 T_default 口径 Q7 待确认。
5. **CVT 已确认**：源类型编码 = Track-B §5.3（dst[6:3]/src[13:10]），已由 `cvtff` 执行 golden 验证。隐藏测试的精度组合覆盖仍受 ISA 类型面约束（无 fp8/fp4/int4 标量）。
6. **多 kernel 未实现**：若隐藏测试有多个 entry 函数，当前只编译第一个。
7. **依赖 `public/` 路径**：`Makefile` 和测试脚本引用 `../public/Track-C/`，需确保路径在各环境一致。
