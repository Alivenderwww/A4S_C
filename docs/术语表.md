## A. 编译器与中间表示

**编译器（Compiler）** — 把一种语言（这里是 PTX 风格的 IR）翻译成机器能跑的代码（AEC 二进制）。赛道 C1 就是写一个编译器。

**IR（Intermediate Representation，中间表示）** — 编译器内部的「中间语言」。源代码先翻成 IR，IR 再翻成机器码。好处是优化可以在 IR 这一层统一做。详见 §6。
> 一句话理解：IR 像是「半成品菜」——先把食材处理成统一的半成品，再决定怎么做成最终菜品（机器码）。

**PTX（Parallel Thread Execution）** — NVIDIA 给 CUDA 设计的虚拟指令集（也是一种 IR）。赛道 C1 的输入 `.ptx` 文件就是这种格式。详见 §6.1。

**SSA（Static Single Assignment，静态单赋值）** — 一种程序写法约定：**每个变量只被赋值一次**，再要改值就换个新名字。控制流汇合处用 φ（phi）函数选值。这样程序特别好分析、好优化。详见 §6.10。
> 一句话理解：变量像「一次性杯子」，用完就换新的，绝不重复用——别人一眼就能看清每个杯子里装的是哪次的值。

**CFG（Control Flow Graph，控制流图）** — 把程序画成图：每个方块是一段顺序代码（基本块），箭头表示「执行完这块可能跳到哪」。编译器靠它分析程序走向。

**基本块（Basic Block）** — 一段**从头到尾顺序执行、中间不跳转**的代码。整个块只有一个入口、一个出口。

**φ 函数（Phi 函数）** — SSA 里用的特殊操作：在 if/else 汇合的地方，根据「刚才走的是哪条路」选一个值。

**支配边界 / 支配树（Dominance Frontier / Tree）** — 控制 CFG 结构的工具，用来决定 φ 函数该插在哪里。对做题来说知道「这是插 φ 的依据」即可。

**Lowering（下降 / 下译）** — 把高层 IR 翻成更底层、更接近硬件的指令。比如把「矩阵乘」这个高层操作，下降成一堆具体的 load/mma/store 指令。

## B. 编译优化（那些 Pass）

**Pass** — 编译器里的一道「处理工序」。源代码流过一道道 pass，每道做一类优化。下面这些都是常见的优化 pass。

**常量传播（Constant Propagation）** — 已知某变量等于一个常量，后面用到的地方直接替换成那个常量。比如 `x=2; y=x+3` → `y=2+3` → `y=5`。

**DCE（Dead Code Elimination，死代码消除）** — 删掉「算出来的结果没人用」的代码，像删掉永远不被读的变量赋值。

**CSE（Common Subexpression Elimination，公共子表达式消除）** — 同样的计算出现多次，只算一次，结果存起来复用。比如 `a=b*c; d=b*c` → 算一次 `t=b*c`，`a=t; d=t`。

**LICM（Loop-Invariant Code Motion，循环不变量外提）** — 循环里每次都算一样的值（不依赖循环变量），就把它搬到循环外面只算一次。

**GVN（Global Value Numbering，全局值编号）** — 给每个计算结果编号，相同的计算得到相同编号，从而发现并合并重复。

**内存合并 / 合并访存（Memory Coalescing）** — GPU 上，让同一时刻相邻线程访问相邻的内存地址，硬件就能把它们**合并成一次大访问**，省时间。详见 §6.3。

**谓词执行（Predication）** — 给指令加个条件，只有条件成立才真正生效。GPU 用它来避免代价昂贵的分支跳转（`@%p` 就是「当 %p 为真时执行」）。

**循环不变量（Loop Invariant）** — 见 LICM，循环里每次都一样的值。

## C. 寄存器分配

**寄存器（Register）** — CPU/GPU 里**最快**的存储位，数量很少（AEC 每线程最多 256 个）。变量想被运算就得先进寄存器。

**寄存器分配（Register Allocation）** — 决定「哪些变量占用哪个寄存器」的过程。寄存器不够时，有些变量得临时请出去。详见 §6.4。

**活跃区间（Live Interval）** — 一个变量「从第一次被赋值，到最后一次被使用」的这段范围。寄存器分配就是看这些区间是否重叠。详见 §6.4.1。

**Linear Scan（线性扫描）** — 一种寄存器分配算法：把所有活跃区间按起点排序，扫一遍，贪心地分配寄存器。简单、快，AEC 推荐用这个。详见 §6.4.1。

**图染色（Graph Coloring）** — 另一种寄存器分配算法：把「同时活跃的变量」连成边，转化成「给图上色、相邻节点不同色」的问题。质量更好但更慢。详见 §6.4.2。

**Spill（溢出）** — 寄存器不够用时，把某个变量的值临时存到慢速的 local memory 里，要用时再读回来。spill 多了程序会变慢，是个重要的性能诊断指标。

**Spill Cost（溢出代价）** — 决定「该把谁请出寄存器」的依据：选代价最小的 spill。

## D. 指令调度

**指令调度（Instruction Scheduling）** — 决定一堆指令**按什么顺序发射/执行**。顺序好，硬件就能更忙、更快。详见 §6.5。

**DDG（Data Dependency Graph，数据依赖图）** — 把指令画成图，边表示「这条指令必须等那条先完成」。

**RAW / WAR / WAW（读后写 / 写后读 / 写后写）** — 三种依赖关系：
- **RAW**（真依赖）：先写后读，必须保序。
- **WAR**（反依赖）：先读后写，不能在读之前改。
- **WAW**（输出依赖）：两次写同一个地方，顺序不能乱。

**List Scheduling（列表调度）** — 经典调度算法：维护一个「前驱都完成了、可以执行」的 ready 队列，按优先级挑一条条发射。详见 §6.5.2。

**双发射（Dual Issue）** — 硬件一拍能同时发射两条指令（前提是它们能配对、不冲突）。AEC ISA 是双发射模型，配对规则是 C1 性能的核心。详见 §6.5.4。

**关键路径（Critical Path）** — 程序里最长的「必须串行」的依赖链。缩短它能直接缩短总时间。

**软件流水线（Software Pipelining）** — 让循环的**不同迭代交错执行**，提高并行度。关键参数 **II（Initiation Interval）** = 每隔几拍启动新一轮迭代。

**延迟（Latency）** — 一条指令从发射到结果可用要等几拍。调度时要避开「还没出结果就用」。

## E. GPU 与 SIMT 执行模型

**SIMT（Single Instruction, Multiple Threads）** — GPU 的执行哲学：**一条指令，很多线程同时各跑各的数据**。和 CPU 的「一条指令一个数据」不同。详见 §6.3.2。

**Warp** — GPU 调度的基本单位，**一组 32 个线程**一起走同一条指令（只是各看各的数据）。Warp 里的线程要么一起走，要么一起等。

**SM（Streaming Multiprocessor，流多处理器）** — GPU 内部一块「计算核心区」，上面跑着很多 warp。一个 GPU 由多个 SM 组成。

**Thread / Block / Grid（线程 / 线程块 / 网格）** — GPU 的三层组织：
- **Thread**：最小单位，一个「工人」。
- **Block**：一组工人（最多 1024 个），被派到一个 SM 上，彼此能共享内存、能同步。
- **Grid**：一次 kernel 启动的所有 block，组成一个「大工程队」。

**Lane** — Warp 里的一个线程位置（lane 0~31）。

**Kernel** — 在 GPU 上跑的那个函数。主机端「启动 kernel」=叫 GPU 去执行这个函数。

**Tensor Core（张量核心）** — GPU 里专门做「小矩阵乘加」的硬件单元，比普通 CUDA Core 快得多。做 GEMM 的主力。详见 §6.3.5、§6.6.3。

**Compute Capability（计算能力版本，CC）** — NVIDIA 给每代 GPU 的版本号（如 7.0 = Volta，8.0 = Ampere），决定它支持哪些指令、多少寄存器。

## F. 内存层次与数据搬运

GPU 内存从快到慢、从近到远分好几层（详见 §6.3.3）：

**Register（寄存器）** — 最快，每个线程私有，数量有限。

**Shared Memory（共享内存）** — 一个 Block 内所有线程共享的小块高速内存，用来在工人之间快速传数据。

**Local Memory（局部内存）** — 名字像本地，其实**很慢**，是寄存器 spill 后的「溢出仓库」。

**Global Memory（全局内存 / 显存）** — 容量大但慢，所有线程都能访问，数据生命周期跨 kernel。AEC 虚拟设备显存 64 MiB。

**Constant / Texture Memory** — 只读、带缓存的特殊内存，分别放常量和纹理数据。

**H2D / D2H（Host-to-Device / Device-to-Host）** — 数据在**主机（CPU）和设备（GPU）之间**搬运的两个方向。

**DMA（Direct Memory Access，直接内存访问）** — 不用 CPU 一步步管，硬件直接搬数据，省 CPU 又快。AEC 虚拟设备有**两个 DMA 通道**（可以同时搬两路）。

**注册内存 / 零拷贝（Registered Memory / Zero-copy）** — 把主机内存「登记」给设备，设备能**直接访问**，免去来回拷贝。小数据用它最划算。

**故障恢复（Fault Recovery）** — DMA 传输出错时能检测、能重试，不死掉。

## G. GEMM 与 Tensor Core

**GEMM（General Matrix Multiply，通用矩阵乘）** — `C = α·A·B + β·C` 这种「矩阵乘再加点东西」的运算。深度学习的核心计算，赛道 C 的性能大头。详见 §6.6。

**BLAS（Basic Linear Algebra Subprograms）** — 线性代数标准接口库。GEMM 属于 **Level-3**（矩阵级运算）。详见 §6.6.1。

**Tiling / 分块（分块计算）** — 把大矩阵切成小块，让小块住进高速缓存/共享内存里反复用，减少慢速访存。详见 §6.6.2。

**Microkernel（微内核）** — GEMM 实现里最内层、最热点的那段计算，通常精心手写、用上寄存器分块和 tensor core。

**mma.sync / wmma / wgmma** — NVIDIA tensor core 的几代指令：
- **wmma**（Volta+）：warp 级的「加载-计算-存储」抽象。
- **mma.sync**（Turing+）：更底层的寄存器级指令。
- **wgmma**（Hopper+）：warp-group 级的大块异步指令。

**TMUL** — 赛题里对「张量乘法指令」的叫法，相当于 AEC 自己的 tensor core 指令。C1 的 PTX-05 要把它 lower 出来。

**CUTLASS** — NVIDIA 开源的 GEMM/Conv 高性能库，是写 GEMM 的事实标准参考。详见 §6.6.4。

**CuTe** — CUTLASS 里的 DSL，把「数据布局」和「计算索引」用代数方式组合，处理分块分区的利器。

**Tile / Tile 大小** — 分块时每块的尺寸（如 `16×8×8`），选对了性能差很多。

## H. 数据精度（一堆 FP/INT）

「精度」指一个数用多少 bit、怎么编码来存。**bit 越少越省显存和算力，但精度越低**。赛道 C2/C3 要在 9 种精度间选。详见 §6.3.5。

| 精度 | 位宽 | 组成（符号-指数-尾数） | 通俗说明 |
|------|------|------------------------|----------|
| **FP64** | 64 | 1-11-52 | 双精度，科学计算用，GPU 上少见 |
| **FP32** | 32 | 1-8-23 | 单精度，深度学习的「标准答案/参考值」 |
| **TF32** | 19（装在 32 位里） | 1-8-10 | Ampere 引入，牺牲一点精度换速度 |
| **FP16** | 16 | 1-5-10 | 半精度，训练/推理常用 |
| **BF16** | 16 | 1-8-7 | Google 版的 16 位，动态范围大、精度略低 |
| **FP8 E4M3** | 8 | 1-4-3 | 8 位，精度优先（尾数多） |
| **FP8 E5M2** | 8 | 1-5-2 | 8 位，动态范围优先（指数多） |
| **FP4 E2M1** | 4 | 1-2-1 | 4 位，极致省空间，精度很差 |
| **INT8 / INT4 / INT32** | 8 / 4 / 32 | 整数 | 定点整数，常用于量化推理 |

**符号 / 指数 / 尾数（Sign / Exponent / Mantissa）** — 浮点数的三个组成部分：
- **符号位**：正还是负。
- **指数位**：决定能表示的**大小范围**（多大、多小）。
- **尾数位**：决定**有效数字的精度**（能精确到几位）。

> 一句话理解 E4M3 vs E5M2：E4M3 给尾数多一位（精度好），E5M2 给指数多一位（范围大）。一个更准、一个更稳。

**混合精度（Mixed Precision）** — 同一次计算里不同步骤用不同精度（比如用 FP16 算、FP32 存累加），兼顾速度和精度。

## I. ONNX 与神经网络算子

**ONNX（Open Neural Network Exchange）** — 一个开放的模型文件格式，让模型能在不同框架间搬来搬去。赛道 C3 的输入就是 ONNX 模型。详见 §6.2。

**DAG（Directed Acyclic Graph，有向无环图）** — ONNX 模型的内部结构：节点是算子，箭头是数据流，顺着箭头走、不会绕回。C3.1 要把 ONNX 解析成这种图。

**Opset（算子集版本）** — ONNX 的「版本号」，每个版本定义一组算子的行为。不同版本同一个算子可能有细微差别。详见 §6.2.3。

**算子（Operator / Op）** — 神经网络计算图里的一个「操作节点」，比如矩阵乘、加法、卷积。C3 评测支持 17 种算子。

常见算子速查（详见 §6.2.3 对应表）：

- **MatMul** — 矩阵乘。
- **Conv / Conv2d** — 卷积，图像处理核心操作。
- **Add / Mul / Div** — 逐元素加 / 乘 / 除（每个位置各算各的）。
- **Softmax** — 把一组数变成「加起来等于 1 的概率」，分类网络末尾常用。
- **LayerNorm（层归一化）** — 把一层的输出调成均值 0、方差 1，稳定训练。Transformer 里必备。
- **BatchNorm（批归一化，BN）** — 按批次做归一化，CNN 里常用。推理时常「折进卷积权重」。
- **ReLU** — 激活函数，把负数变 0、正数不变。
- **GELU** — 另一种激活函数，Transformer 里常用；在 ONNX 里被拆成 `Div + Erf + Add + Mul` 四步。
- **Gemm** — 「广义矩阵乘」`C = αAᵀBᵀ + βC`。
- **Erf（误差函数）** — 一个数学函数，GELU 的组成部分。
- **Reshape / Transpose / Flatten / Split / Gather** — 形状变换类算子（改维度、转置、展平、切分、按索引取值）。
- **GlobalAveragePool** — 全局平均池化，把整个特征图压成一个数。

## J. 算子融合

**算子融合（Operator Fusion）** — 把几个相邻的小算子**合并成一个大算子**，省掉中间结果来回读写显存。是提升推理速度的关键手段。详见 §6.7。

> 一句话理解：与其「算一步存一下、再读再算下一步」，不如「一口气算完中间不落地」，省下大量读写。

**预融合（Pre-fusion）** — 在主融合 pass 之前先做的「准备性融合」。比如 ResNet 的 BN 已经折进卷积了（图里看不到 BN 节点），要先把它显式「展开」回来，才能命中 `FusedConv2dBatchNorm` 这个 pattern。

**BN 折叠（BN Folding）** — 推理时，BatchNorm 的数学运算可以**直接算进前面卷积的权重和偏置**里，省掉一个独立算子。公式见 §6.7.4。

**im2col** — 把卷积变成矩阵乘的技巧：把图像的每个感受野**展平拼成矩阵的一列**，卷积就成了普通矩阵乘。详见 §6.7。
> 一句话理解：把「滑窗卷积」改写成「大矩阵乘」，这样就能复用高效的 GEMM 内核。

**Winograd** — 另一种加速卷积的算法，用**更多加法换更少乘法**，对特定小卷积核特别快。C3.2 要求 im2col 和 Winograd 都被用过。详见 §6.7。

**Fusion Pattern（融合模式）** — C3.3 要命中的 5 种目标融合：`FusedMatMulBias`（矩阵乘+偏置）、`FusedConv2dBatchNorm`（卷积+批归一化）、`FusedEWChain`（逐元素链）、`FusedSoftmaxDropout`（Softmax+Dropout）、`FusedResidualNorm`（残差加+层归一化）。

## K. 内存规划

**内存规划（Memory Planning）** — 决定程序运行时**每个中间结果存在哪、什么时候可以回收复用**。规划得好，显存占用大幅下降。C3.4 的核心。详见 §6.8。

**生命周期（Lifetime）** — 一个中间张量「从第一次产生到最后一次被用」的时间区间。区间不重叠的两个张量可以共享同一块显存。详见 §6.8。

**DSA（Dynamic Storage Allocation，动态存储分配）** — 内存规划背后的理论问题：给一堆有起止时间的 buffer 分配位置，使总占用最小。已被证明是 NP 完备。详见 §6.8.1。

**区间图染色（Interval Graph Coloring）** — 当所有 buffer 同尺寸时，DSA 退化成这个问题，有高效的贪心解法。

**内存池（Memory Pool）** — 预先申请一大块显存，后面的小申请都从池子里切，避免反复申请释放的开销。

**权重预取（Weight Prefetch）** — 趁当前层还在算，**提前把后面几层的权重搬过来**，算完就能立刻接上，「边算边传」。

**流级并行（Stream-level Parallelism）** — 把没有依赖关系的算子分到不同的「流」上，让它们并行跑。

## L. 工具链与开源框架

**LLVM** — 一套著名的编译器基础设施，很多编译器（包括 Clang、Rustc）都基于它。AEC 编译器可参考它的思路，但**别真硬上 LLVM**（指令子集小，手写更快）。详见 §6.9。

**MLIR（Multi-Level Intermediate Representation）** — LLVM 团队做的「多层级 IR 框架」，能在同一套基础设施里表达从高层算子到机器码的多层 IR。自建编译器骨架的低成本路径。详见 §6.9。

**Dialect（方言）** — MLIR 里的「IR 模块包」，比如 `gpu`、`affine`、`linalg`、`llvm` 各管一摊。选手可以自己写一个 `aec-isa` dialect。

**Progressive Lowering（渐进下降）** — MLIR 的理念：高层算子**一层层往下降低**（vector → affine → llvm → 机器码），每层做各自的优化。

**TVM** — Apache 的深度学习编译器，里面的算子融合算法（FuseOps）是 C3.3 的理论参考。详见 §6.7.2。

**XLA** — Google 的深度学习编译器，它的 `BufferAssignment` 是 C3.4 内存规划的理论参考。详见 §6.8.2。

**Triton** — OpenAI 开源的 GPU 编程语言/编译器，写 GPU kernel 像写 NumPy，自动管 tile 和内存。它的 AutoWS 算法是 C3.4 的参考。详见 §6.8.4。

**cuDNN** — NVIDIA 的深度学习算子库，里面定义了一套官方的融合 pattern。详见 §6.7.3。

**cuBLAS** — NVIDIA 的 BLAS 实现（矩阵运算库），它的 GEMM 接口形状可以对照 C2 的 10 个 `aecMatmul*`。

## M. AEC 竞赛专有名词

**AEC / AEC ISA** — 本赛题自创的 GPU 指令集架构。指令是 **128-bit 定长**编码，支持双发射。一切以赛题提供的规范 + `golden/b_isa_public.json` 为准。

**AECIR** — 赛题给编译器输入的「PTX 风格 IR」，C1 把它编译成 AEC 机器码。

**aecbin** — AEC 的二进制可执行文件格式（Header + Code Section + Data Section + Relocation + Symbol Table）。C1 编译器的输出。

**aec-cc** — 选手要实现的 AEC 编译器命令：`aec-cc input.ptx -O{0|2|3} -o output.aecbin`。

**aec-objdump** — 选手要实现的反汇编工具，把 aecbin 还原成可读指令。

**虚拟设备 / 虚拟周期（Virtual Device / Virtual Cycle）** — C2 用一个**确定性的模拟设备**来跑程序，性能不按真实时间，而按「虚拟周期数」算。这样评分可复现、不挑硬件。

**AEC_DEVICE_OP_ISA_LAUNCH** — 启动 kernel 的核心设备操作。C2 强制所有 kernel 必须走这条路径，**禁止在主机端偷偷算**。

**TMUL 指令** — AEC 的张量乘法指令（对标 NVIDIA tensor core 的 mma）。C1 要把它从 IR lower 出来。

**34 个固定 image** — C2 给定 34 个预编译好的 kernel 镜像，选手**只能从中选、不能自己造**。

**Starter Kit** — 赛题提供的「起步包」，含头文件、示例代码、公开测试、评分脚本等。C2 的最完整。

## N. 评测与得分相关

**门禁 / 门槛（Threshold）** — 评测的硬性及格线，达不到直接 0 分。比如 C3.5 要求 `max_abs_diff ≤ 1e-3`，MLP 准确率 ≥ 98%。

**正确性门禁** — C1 里「先算对才能谈性能」的规则：程序跑不对，性能分免谈。

**性能归一化（Performance Normalization）** — 把你的成绩和基线比，换算成 0~100 分的公式。C1 用 `r = T_base / T_你`，再过一个对数 clip。详见 `赛道C-知识库与作战手册.md` §0。

**几何平均（Geometric Mean）** — 一种平均方式（把各比值连乘后开 N 次方），用来综合多项性能比。C1 的 Agent 评分用它。

**Agent** — 赛题里的「智能调度程序」。C2 有两个：**DMA Agent**（决定怎么搬数据）和 **Kernel Agent**（从固定镜像里选哪个）。C1 也有 Agent 做自动优化。Excellent 等级必须有 Agent。

**Golden Model / Golden Logits** — 评分用的「标准答案」。你的输出要和 golden 对齐才算对。

**Cycle Model（周期模型）** — 估算一段代码要跑多少周期的模型。C1/C2 的性能分都基于周期数。

**诊断指标（Diagnostic Metric）** — 不计分、但用来调试的性能数据，比如 `spill_count`、`dual_issue_rate`、`memory_transactions`。

**Code Review（代码审查）** — C3.4 用人工/脚本读你的代码来评分（不跑测试）。**只写注释或空壳接口不得分**，必须真实落地算法。

**变异测试（Mutation Testing）** — 拿原始测试做各种改动（换参数、换寄存器、打乱顺序、换数据类型）看你的实现还能不能对。C1 的鲁棒性分（5 分）考这个。
