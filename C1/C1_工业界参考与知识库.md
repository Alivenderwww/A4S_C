# C1 工业界参考与知识库

> 目的：C1 编译器的每个阶段"别闭门造车"，对照工业界/开源的成熟做法。
> 按**我们的 pipeline 阶段**组织，每条给出：参考来源 → 核心技术 → **对 AEC C1 怎么用（借什么/差异在哪）**。
> 开发期可联网查阅这些（评测才离线）。用到的都记入原创性声明。

## 0. 整体架构：直接对标 ptxas（PTX→机器码，和我们同构）

- **ptxas 逆向参考**（[gh.evko.io/nvopen-tools/ptxas](https://gh.evko.io/nvopen-tools/ptxas)）：NVIDIA 自己的 PTX→SASS 汇编器，**159 阶段** pipeline，含 copy-prop、CSE、strength reduction、**predication**、rematerialization、**GMMA pipelining**、late legalization，**Fatpoint 寄存器分配器**、**3 阶段指令调度器**，最后 Mercury 编码器（~4000 编码 handler + 3 个 peephole）。
  - **对我们**：这就是我们任务的"完整版蓝图"。它的 pass 列表**验证了我们的 pipeline**（const-prop/CSE/DCE/LICM/预测化 + 调度 + 分配），并提示可加：**strength reduction**（`x*4`→`x<<2`）、**rematerialization**（重算比 spill 便宜的值）。PTX↔SASS 非一一对应（一条 PTX 可展开/消除/融合）——我们也不该逐条直译。
- **LLVM NVPTX / MachineScheduler**（见 §5、§6）：开源、可读的后端实现，作为我们 regalloc/调度的工业参考。
- **Accel-Sim / GPGPU-Sim**（[github](https://github.com/accel-sim/gpgpu-sim_distribution)）：功能 + cycle 模拟器；Accel-Sim 支持 vISA(PTX) execution-driven 和 SASS trace-driven。**我们的 `sim/` 是它的迷你版**；cycle 模型的延迟结构可借鉴（见 §9）。

## 1. 标量优化（T2：const-prop / CSE / DCE / LICM）—— 教科书 + ptxas 验证

- ptxas pipeline 里就有 copy-prop / CSE；我们的 T2 是标准 SSA 数据流。
- 参考：经典编译原理（到达定义、可用表达式、循环不变量检测）；先建**支配树 + 自然循环识别**（扩展 `cfg.cpp`）再做 LICM。
- **对我们**：PTX-02 里 `%f5==%f6`(CSE)、`%f15` 死值(DCE)、`a+b`/`a*b` 提出 32 次循环(LICM)。可再加 **strength reduction**（地址算 `idx*4`→移位）。

## 2. 谓词化 / if-conversion（T2 / §1.3）—— 工业界有明确"边界"

- 关键结论（多方一致，[SIMT 分歧综述](https://arxiv.org/pdf/2107.05681v1)、CUDA divergence 文档）：**只对"小分支"做谓词化**（编译器算并管理 active lane-mask）；**大的发散分支保留为真分支**（硬件管理 mask），否则两路都取指反而更慢。
- **对我们**：AEC 无重汇聚栈，越界 guard 是**小分支** → 谓词化正合适（我们 §1.3 方向正确）；但别把大循环体也无脑谓词化。uniform 循环回边保留 BRX。

## 3. 寄存器分配（T4）—— ptxas Fatpoint / LLVM

- ptxas 用 **Fatpoint** 分配器；LLVM 用图着色/贪心 + **调度在分配前降低寄存器压力**。
- **活跃度感知的线性扫描**（Poletto&Sarkar，[UCLA PDF](http://web.cs.ucla.edu/~palsberg/course/cs132/linearscan.pdf)）。**工业参考**：真 spill（选最远 end 区间→LMEM slot→插 LD/ST），可参考 LLVM 的 spill/coalesce。**关键顺序**：**先调度后分配**（pre-RA scheduling，见 §6）能减压。

## 4. 指令调度 + 双发射（T4）—— SASS 控制码是最佳模板

历史设计目标（`list_sched.cpp`）：DDG 构建（RAW/WAR/WAW + 延迟表）→ 关键路径高度优先级 → 就绪列表调度 → 双发射计数。当前实现状态见 [`../docs/C1-完成度审计.md`](../docs/C1-完成度审计.md) §三。工业参考：**增强与官方参数校准**：

- **NVIDIA SASS 控制码**（[DocumentSASS](https://github.com/0xD0GF00D/DocumentSASS)、[Kuter Dinel 控制码](https://kuterdinel.com/nvidia-sass-control-code-viewer.html)、["Analyzing Modern NVIDIA GPU cores"](https://www.alphaxiv.org/overview/2503.20481v1)）：Maxwell 起 NVIDIA 把调度**静态编码**进每条指令：`wait barrier mask : read/write barrier mask : yield : stall count`。
  - **fixed-latency 指令**（算术）→ **stall count 硬编码**（发下一条前等几拍）。
  - **variable-latency 指令**（global load / SFU）→ 用 **6 个 scoreboard barrier**（写时置位、消费者 wait）。
  - 调度策略 **CGGTY**（Compiler Guided Greedy Then Youngest）：优先同 warp 就绪指令。
  - **对我们**：AEC 的 LD/LMEM 是 32-cycle（Track-B §7），正是 variable-latency。我们的 list scheduler 应：**把独立 load 提前发射**（一次性发多个，靠 outstanding≤16 隐藏 32 拍），再穿插算术；**双发射**=把无依赖指令配对（VLIW 式静态打包）。cycle 模型应：消费者若在 load 完成前使用就**计 stall**。
- **LLVM MachineScheduler**（[ScheduleDAGMILive](https://llvm.org/doxygen/classllvm_1_1ScheduleDAGMILive.html)、[Min Hsu 教程](https://myhsu.xyz/llvm-machine-scheduler/)、[VLIWMachineScheduler](https://llvm.org/doxygen/classllvm_1_1VLIWMachineScheduler.html)）：工业级 DAG list scheduling 参考实现，带**寄存器压力跟踪**；核心：build DDG → ready list（priority=关键路径/延迟）→ 逐条发射。**明确结论：调度放在寄存器分配前，用于降低寄存器压力。**
- **List scheduling / VLIW 基础**（[CMU 15-740 VLIW](https://www.cs.cmu.edu/afs/cs/academic/class/15740-f18/www/lectures/17-18-vliw.pdf)）：ready-list + 优先级；**软件流水**（software pipelining）重叠循环迭代隐藏延迟——对 PTX-03/05 的循环很有用。
- **对我们（list_sched.cpp 实现要点）**：① 建 DDG（真/反/输出依赖）；② 优先级用关键路径长度（含 load 32 拍权重）；③ 贪心 list scheduling；④ 双发射：扫描相邻就绪对，无依赖且端口不冲突则配对；⑤ 先于 regalloc 跑。

## 5. GEMM → TMUL 下降 + tiling（T5，最大奖 16+11）—— Siboehm/CUTLASS 是路线图

- **Siboehm《How to Optimize a CUDA Matmul Kernel》**（[siboehm.com/articles/22/CUDA-MMM](https://siboehm.com/articles/22/CUDA-MMM)）：naive→cuBLAS 95% 的**逐步 tiling 路线**，是单篇最佳参考。层级：
  1. naive → 2. **全局访存合并**（同 warp 连续地址合并成一次事务）→ 3. **SMEM 分块缓存**（BM=BN=BK=32，把 A/B tile 载入 SMEM 复用，SMEM ~12TB/s vs GMEM 750GB/s）→ 4. **1D 寄存器分块**（每线程算 TM=8 个输出，提高算术强度）→ 5. **2D 寄存器分块**（TM×TN=8×8，外积式累加）→ 6. **向量化访存**（float4/LDS.128，A 在 SMEM 转置）→ 9. **autotune tile 尺寸** → 10. **warptiling**（block→warp→thread 三级）。
- **CUTLASS/CuTe**（[efficient_gemm.md](https://github.com/NVIDIA/cutlass/blob/main/media/docs/cpp/efficient_gemm.md)、[Colfax GEMM 教程](https://research.colfax-intl.com/cutlass-tutorial-design-of-a-gemm-kernel/)）：层级 tiling（block/warp/thread）+ **SMEM staging** + **双缓冲/软件流水**（一块 tile 算、另一块预取）+ warp 级 **MMA**。
- **Triton `tl.dot`**（[OpenAI Triton](https://openai.com/index/triton/)）：tile 抽象自动做合并/SMEM/同步；FP8/FP16 GEMM 开箱即用。**tritonBLAS**（[arxiv](https://arxiv.org/html/2512.04226v1)）：**解析式**选 GEMM tile 参数（比搜索便宜）——对我们的 Agent 调 tile 尺寸有用。
- **对我们（gemm_tmul.cpp 实现要点）**：① 识别 K-loop MAC 惯用法（历史 detection 脚手架设计）；② **tile 化**：M/N 按 16×16、K 按 BK 分块；③ **SMEM staging**：`LD.gmem`→`ST.smem` 把 A/B tile 载入 smem（AEC 有 65536B/CTA smem），循环内从 smem 读；④ 每 tile 发 `TLDA`(载 A tile)+`TLDA`(载 B tile)+`TMUL`(mode 按 dtype：f16=1)+累加，末尾 `TSTA` 写回；⑤ **多精度参数化**（fp4/fp8/bf16/f32/int，TMUL mode 表见 docs/03）；⑥ **非 16 倍数余数 tile 用谓词/掩码**；⑦ 双缓冲预取（进阶）。tile 尺寸交给 Agent 搜。

## 6. Agent 自动调优（D，10 分）—— TVM Ansor/AutoTVM 范式

- **TVM AutoTVM / Ansor**（[Ansor 论文](https://arxiv.org/pdf/2006.06762)、[intro](https://tvm.apache.org/2021/03/03/intro-auto-scheduler)）：编译器自动调优的工业范式 = **cost model（代理）+ 搜索**。AutoTVM=模板式（特征向量：loop extent、访存次数 → 预测）；Ansor=无模板 sketch 生成 + **进化搜索** + 学习型 cost model（XGBoost）。真实测一次要 ms，cost model μs 级剪枝。
- **tritonBLAS**：**解析式**参数选择（轻量，不搜）。**CuAsmRL**（[arxiv](https://arxiv.org/pdf/2501.08071)）：RL 调 SASS 调度（前沿）。
- **对我们**：Agent 循环 = 在**配置空间**（-O 级别、tile 尺寸 BM/BN/BK、unroll、是否 smem 提升）上，用 **本地 cost model + 搜索**（网格/贪心/进化）找最优；读性能报告→重编译→验证→生成报告（闭环 2 分）。目标 geomean ≥1.25×。**离线约束**：评测时 Agent 不能连网，cost model 须本地自足。当前代理实现状态见 [`../docs/C1-完成度审计.md`](../docs/C1-完成度审计.md) §六。

## 7. 功能 / cycle 模拟器（我们的 oracle）—— GPGPU-Sim/Accel-Sim + SASS 延迟

- `sim/` 功能正确性 oracle（回归结果见 [`../docs/C1-完成度审计.md`](../docs/C1-完成度审计.md) §二）。**cycle 模型**要更真实时，借鉴：
  - SASS 延迟结构（§4）：fixed-latency 硬编码拍数 + variable-latency scoreboard；LD/LMEM 32 拍、outstanding≤16、128B line（Track-B §7）。
  - [GPGPU-Sim](https://github.com/gpgpu-sim/gpgpu-sim_distribution) 的 warp scheduler / scoreboard / mem 模型可作参照。
- **对我们**：在 sim 上叠一层简单 cycle 模型（每指令基础拍 + load 未就绪时 stall + 双发射减 1 拍），让 Agent 的 `est_cycles` 和真 cycle 相关性更好。

## 8. Performance Model 背景

> Performance Model 的唯一事实源（包括当前代理口径详情、缺失参数）见 [`../docs/C1-完成度审计.md`](../docs/C1-完成度审计.md) §六，本文仅提供工业参考背景。

AEC 的 Performance Model 由 Track-B `spec.md` 约束基本参数（warp=32, GPR=256, SMEM=64KB, LMEM=4KB, LD latency=32 cycles, outstanding≤16），但**未提供完整延迟表或可执行 C-model**（属赛道 B 交付物）。

工业参考（可在开发期查阅；评测时离线）：
- **SASS 控制码模型**（§4）：NVIDIA 把调度静态编码进每条指令（wait barrier mask · stall count），是 C1 list scheduler + scoreboard 的工业参照。
- **[GPGPU-Sim](https://github.com/gpgpu-sim/gpgpu-sim_distribution)**：其 warp scheduler / scoreboard / mem 模型的延迟结构可借鉴。
- **SM 数量未公开**：只有官方 total_cycles 按完整 grid 并行建模时才可能影响，当前本地代理未建模该因素，实际口径待确认。
- 若组委会后续发布二进制参考 C-model，应以此为准替换所有代理口径。
