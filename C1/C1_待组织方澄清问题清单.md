### Q2. `.aecbin` 二进制容器的字节级格式与"格式合规性"校验内容

- **引用**(`C1-compiler/spec.md` §2 输出):
  > 二进制必须至少包含:
  > `Header` / `Code Section` / `Data Section` / `Relocation Section` / `Symbol Table`
  > Code Section 由 128-bit 定长 AEC ISA 指令组成。
- **引用**(`C1-compiler/scoring.md` A 评测流程):
  > 二进制验证检查格式合规性
- **事实**:"至少包含"这些段,但公开文档未给出 **Header 的字段顺序/大小/magic、段表、Relocation Section 的条目格式、Symbol Table 的条目格式** 等字节级 layout。
- **问**:
  1. "格式合规性"具体校验哪些字段?
  2. Golden Model 的加载器按什么字节布局解析 Header 与各 Section?(指令 128-bit 编码已按 **Track-B §A.1** 对齐并用 Track-B `aec_cases/program.bin` 验证;不确定的只是**容器外壳**。)
- **【组织方答复 2026-07-13,已答】**:「你们可以自己不加这个东西…只需要按照它那个 isa 提供那一段输出就行了,其他格式啥的可以不用太在意」;追问 Header/Code/Data/Relocation 能否置空 →「可以先置为空,先不用管」。
  → **状态:风险大幅下降**。只需保证 Code Section(128-bit 指令)正确,组织方负责映射到 GPU。
  → **仍存二义(见 P0.5-A)**:「不加」倾向裸指令流、「置为空」倾向保留空壳容器;且「先不用管」= 后续可能补规范。参赛方需要一个确定的最小字节契约。

## P1 — 大幅影响得分

### Q4. IR 方言不一致:`spec.md` 示例 vs 真实公开测例

- **引用**(`C1-compiler/spec.md` §3 IR 格式,示例):
  > `.version 1.0`
  > `.target aec_sm_10`
  > `.kernel vector_add(`
  > `st.gmem.f32 [%rd9], %f9;`
- **事实**(`C1-compiler/testcases/PTX-01_vector_add.ptx` 实际内容):
  > `.version 7.0`
  > `.target sm_70`
  > `.visible .entry vector_add(`
  > (访存写作 `st.global.f32`)
- **问**:隐藏测试使用哪一套方言?编译器是否需**同时**接受 `.kernel` + `st.gmem` 与 `.visible .entry` + `st.global` 两种写法?
- **【组织方答复 2026-07-13,已答】**:「c1 输入是英伟达的 ptx…会指定一个版本…第一段那个 acir 那个东西你们可以先不用管」;「马上会发一个 announcement」。
  → **状态:已定**。输入是**固定版本的真实 NVIDIA PTX**(即 testcases 的 `.version 7.0` / `.target sm_70` / `.visible .entry` / `st.global` 风格);`spec.md §3` 的 `.kernel` / `st.gmem` 示例**作废**。**仍需等 announcement 确认具体 PTX 版本号**(见 P0.5-B)。

### Q5. TMUL tile 寻址契约(C1 需自行**生成**,而非执行固定 image)

- **引用**(`C2-runtime/starter-kit/docs/03_AEC_ISA规范.md` §6 Tensor 形式):
  > `TLDA.type R32,[R0]`
  > `TLDA.type R48,[R2]`
  > `TMUL.type R64,R32,R48,R64`
  > `TSTA.type [R4],R64`
  > naive/tiled/vectorized variant 位于 image header,不写入指令字段。
- **引用**(`C1-compiler/spec.md` §4.2):
  > Tile 大小自动调整
- **事实**:上述形式面向 C2 的**固定 image**,地址寄存器 `R0/R2/R4` 为常量,且该文件明确"学生实现 Runtime 和 Agent,不需要生成或修改 kernel image"。C1 却必须**生成** GEMM 代码。
- **问**:C1 生成 TMUL-based GEMM 时,Golden Model 期望的 **tile 循环结构** 与 `R0/R2/R4` 跨 tile 的**地址推进方式**是什么?device 如何按 `M/N/K` 推进各 tile?(仅凭固定 image 无法还原编译器应生成的寻址序列。)
- **【组织方答复 2026-07-13,方向已定 / 细节未决】**:「我们最后要兼容赛道 B…我们的 op code 会转成赛道 B 格式,如果赛道 B 不支持,你们可以不用做…就不会出现在我们 test case 里面」;评分「只有两个,正确 + 快不快」。周旭林追问「没有 TMUL 就只能转成标量做法?」→ 组织方「在跟 B 的同学同步」。
  → **状态:标量 GEMM 路线被验证**——若 Track B 无 TMUL,则 T5 不会考 TMUL,标量 lowering 合法。**TMUL 的 opcode / 是否要求走张量层仍未决**(组织方与 B 组同步中,见 P0.5-C)。这直接支持我们此前"不赌 TMUL"的稳妥决策。

### Q6. 9 种 GEMM 精度是否为 T5 正确性门禁,以及参考数值规则是否适用于 C1

- **引用**(`C1-compiler/scoring.md` §8):
  > PTX-05 的隐藏测试覆盖全部 9 种 GEMM 精度:
  > FP4 E2M1、FP8 E4M3、FP8 E5M2、FP16、BF16、FP32、INT4、INT8、INT32
- **引用**(`C2-runtime/starter-kit/docs/04_数值规范.md` §2 数据格式 / §5 比较规则):
  > F16 / BF16 / F32 的 accumulator 均为 `FP32`;I4/I8/I32 为"精确整数和"
  > `FP32 GEMM  atol=1e-5, rtol=2e-5`
- **问**:
  1. `04_数值规范.md` 属于 C2 Runtime API;其累加/舍入/容差规则是否**同样适用于 C1 的 PTX→AEC GEMM** 正确性判定?
  2. T5 的 20 个隐藏正确性用例是否要求**全部 9 种精度**都通过?还是部分精度不达标仍可拿到 T5 的其余分?
- **【组织方答复 2026-07-13,部分】**:评分「只有正确 + 快不快」;并且「如果赛道 B 不支持…就不会出现在我们 test case 里面」。
  → **状态:精度范围受 Track B 支持面约束**——Track B 支持不了的精度不会考。但「9 精度是否为门禁 / `04_数值规范.md` 是否适用于 C1」仍未正面答复,建议随 P0.5-C 一并确认。

### Q9. 转换指令(CVT)的"源类型"编码位置 —— ✅ 已解决

- **状态**:无需再问。编码器/解码器已按此实现并验证。

### Q10. 寄存器约定:R0 是否保留、64-bit 是否必须占用连续对

- **引用**(`C2-runtime/starter-kit/docs/03_AEC_ISA规范.md` §1):
