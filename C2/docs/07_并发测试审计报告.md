## 1. 概述

本文件审计 `C2/tests/test_runtime_concurrency.py` 中的并发与生命周期测试。重构的核心目标是将**假绿**（race-condition-based 测试因运气而非正确性通过）替换为**确定性**证据链。

### 1.1 测试分类

| 类别 | 测试 ID | 证据类型 | 平台要求 |
|---|---|---|---|
| **ABA 句柄重用** | A1, A2 | 确定性（256 周期暴力检测，无 reuse 则逐操作验证 INVALID_HANDLE） | 同 libaec.so |
| **Destroy 等待** | B1 | 确定性（shim 阻塞 submit → destroy 不得返回 → release → 验证数据） | Linux + g++ |
| **Free 等待** | B2 | 确定性（shim 阻塞 submit → free 不得返回 → release → post-free sync INVALID_ADDRESS） | Linux + g++ |
| **Unregister 等待** | B3 | 确定性（shim 阻塞 submit → unregister 不得返回 → release → zc delta = 0） | Linux + g++ |
| **Event marker** | B4 | 确定性（shim 阻塞 submit → EventRecord 队尾等待 → release → elapsed=0 → destroy） | Linux + g++ |
| **Multi-stream 压力** | C1 | 压力证据（Barrier 起跑，stats 计数，不 claim 线性化） | 同 libaec.so |
| **Same-stream 压力** | C2 | 压力证据（同上） | 同 libaec.so |

**B 类测试**（shim 确定性）在非 Linux／无 g++ 环境下清晰 `SKIP`，不影响总分。

### 1.2 总远端时间保证

所有测试合计 `< 45 秒`（ABA 256 周期 < 25 s，B 类每项 < 25 s，C 类 < 25 s，远端串行运行）。

## 2. 架构决策

### 2.1 为什么不用纯 race / barrier 测试？

旧的 race 测试（Barrier 起跑 + 线程竞争）依赖**时序运气**：

- Destroy 线程在 enqueue 之后调用，但无法保证 Destroy 的**内部等待**已观察到 enqueue 的 submit；
- Free／Unregister 同理；
- 即使测试以概率 99% 通过，剩下的 1% 是假绿 — 测试通过**不是因为** Runtime 正确等待，而是因为调度恰好让 enqueue 先完成。

**解决方案**：LD_PRELOAD shim (`blocking_submit_shim.so`) 拦截 `aecDeviceSubmit`，在调用真正设备前阻塞调用线程，直到测试显式 release。这样测试可以**观察中间状态**（Destroy 尚未返回、Free 尚未返回等），然后 release 验证线性化完成。

### 2.2 Shim 架构

```
  [Worker Thread]         [Test Thread]          [Device]
       │                       │                    │
       │ aecCopyAsync()        │                    │
       │ → aecDeviceSubmit()   │                    │
       │   │                   │                    │
       │   ├─[SHIM: armed?]────┤                    │
       │   │  YES → BLOCK      │                    │
       │   │                   │ aecStreamDestroy() │
       │   │                   │ → (blocks, submit  │
       │   │                   │   still in flight) │
       │   │                   │                    │
       │   │                   │ aecTestRelease()   │
       │   ├─[SHIM: released]  │                    │
       │   │                   │                    │
       │   └─→ real submit ────┼───────────────────→│
       │ ← SUCCESS             │                    │
       │                       │ ← destroy returns  │
```

Shim 导出的 C 函数（可通过 Python ctypes 调用）：

| 函数 | 作用 |
|---|---|
| `aecTestArmSubmitBlock()` | 布防：下次 `aecDeviceSubmit` 进入时阻塞 |
| `aecTestWaitSubmitBlocked(ms)` | 等待线程阻塞在 submit 内，超时返回 0 |
| `aecTestReleaseSubmitBlock()` | 释放阻塞线程，恢复透明转发 |
| `aecTestResetSubmitBlock()` | 撤防 + 重置状态 |

默认状态完全透明，无性能开销。

### 2.3 与非 shim 测试的关系

| 方面 | C1/C2（压力） | B1-B4（确定性） |
|---|---|---|
| 证明目标 | Runtime 不崩溃、不丢命令 | Runtime 满足线性化语义 |
| 时序依赖 | 有（Barrier 起跑，但内部排序不确定） | 无（shim 强制排序） |
| 证据强度 | "在多线程压力下工作" | "在任何调度下满足规范" |
| 措辞 | "pressure evidence" | "deterministic proof" |

C1/C2 不 claim 线性化，只提供压力证据。

## 3. 详细测试说明

### 3.1 A1 — Stream ABA 256 周期 + 过期句柄验证

**协议**：

1. 创建 stream，记录指针值 `s1_val`；
2. Destroy s1；
3. 循环 256 次：create → record pointer value → destroy（保持每个句柄引用）；
4. 若任一 `create` 返回的指针 == `s1_val` → **FAIL**（contract unsatisfiable）；
5. 若无 reuse → 在第一个过期句柄上分别调用 `aecStreamSync`、`aecStreamDestroy`、`aecCopyAsync`、`aecEventRecord`，全部预期 `INVALID_HANDLE`。

**确定性证据**：暴力枚举 256 周期检测 reuse；若无 reuse，每个操作独立验证。无 race condition。

### 3.2 A2 — Event ABA 256 周期 + 过期句柄验证

**协议**：同 A1，但针对 event handle。过期操作包括 `aecEventQuery`、`aecEventSynchronize`、`aecEventDestroy`、`aecEventRecord`、`aecEventElapsedCycles`。

### 3.3 B1 — Destroy 等待 pending submit（shim 确定性）

**协议**：

1. Worker 线程：arm shim → `aecCopyAsync(H2D)` → 阻塞在 `aecDeviceSubmit` 内；
2. 测试线程：等待 `aecTestWaitSubmitBlocked(5000)` 确认 worker 已阻塞；
3. 测试线程：启动 destroyer 线程执行 `aecStreamDestroy`；
4. 等待 200ms，检查 destroy 尚未返回 (destroy_result == -1)；
5. Release submit；
6. 验证：enqueue SUCCESS，destroy SUCCESS；
7. 验证：`aecCopyD2H` 读取数据一致。

**关键断言**：destroy 在 submit 释放前不得返回。这是假绿的主要来源 — 旧测试无法保证这一点。

### 3.4 B2 — Free 等待 pending submit（shim 确定性）

**协议**：

1. Worker 阻塞在 submit；
2. 测试线程：启动 `aecFree` 线程，验证 200ms 未返回；
3. Release submit；
4. 验证 enqueue SUCCESS，free SUCCESS；
5. StreamSync 接受 SUCCESS 或 INVALID_ADDRESS（线性化结果）；
6. **后条件**：对已 free 地址再次 `aecCopyAsync` → SUCCESS（stream 有效）；`aecStreamSync` → INVALID_ADDRESS。

**关键断言**：free 后 enqueue SUCCESS + sync INVALID_ADDRESS 序列。旧 free 测试无法保证 free 期间的 submit 已等待。

### 3.5 B3 — Unregister 等待 pending submit（shim 确定性）

**协议**：

1. 注册 host buffer → worker 阻塞在 submit（使用 registered host ptr）；
2. 测试线程：启动 `aecHostUnregister`，验证 200ms 未返回；
3. Release submit → copy SUCCESS，unregister SUCCESS；
4. 验证 sync SUCCESS；
5. **后条件**：reset stats → 第二次 copy 使用原 host ptr → sync → `zero_copy_commands` 增量 == 0。

**关键断言**：unregister 后，原 host ptr 的 copy 不再使用 zero-copy。旧测试虽有 zc delta 检查但无法保证 unregister 期间 submit 已等待完成，可能误判。

### 3.6 B4 — Event marker 与 pending submit（shim 确定性）

**协议**：

1. Worker 阻塞在 submit（DMA copy 写入 `alloc1`）；
2. 测试线程：`aecEventRecord(ev, stream)` — 应在 stream tail 等待 worker；
3. 验证 200ms 未返回；
4. Release submit → worker 完成 → Record 完成；
5. 立即 record `end_ev`（无中间命令）；
6. Sync both events；
7. `aecEventElapsedCycles(ev, end_ev) == 0`（同一 stream position）；
8. `aecCopyD2H` 验证 DMA 数据正确；
9. `aecEventDestroy(ev)` 和 `aecEventDestroy(end_ev)` SUCCESS。

**关键断言**：
- EventRecord 在 submit 释放前不返回（正确作为 stream tail）；
- elapsed=0 证明 ev 和 end_ev 在同一 stream point（re-record 语义正确）；
- EventDestroy 在 latest generation 上成功。

### 3.7 C1/C2 — 压力测试

**协议**：传统 Barrier 起跑 + 线程竞争。多 stream（C1）和单 stream（C2）。

**措辞**：所有 `NOTE` 输出和文档明确标注 "pressure evidence"（压力证据）。不 claim 线性化。

## 5. 局限性

- **Shim B 类测试**仅限 Linux + g++。Windows/WSL 用户需要 g++ 环境；非 Linux 平台自动 SKIP。
- **ABA 测试**依赖 allocator 行为。若 allocator 永远不 reuse 指针则测试永远 PASS（这不是假绿，而是 contract 在该 build 上可实现）。
- **压力测试**无法覆盖所有交错。它们不证明线性化正确性 — shim 测试负责这部分。
- 远端环境需 `libaec_device.so` 和编译后的 `libaec.so`。

## 6. 远端红测修复（v2）

### 6.1 Shimming 编译修复

`blocking_submit_shim.cpp` 添加 `<unistd.h>`（提供 `_exit`）和 `<cerrno>`（提供 `ETIMEDOUT`），确保 Linux g++ 编译通过。

### 6.2 提交线性化 mutex

**问题**：Same-stream 并发压力测（C2）中，两线程同时 `aecDeviceSubmit`，`fetch_add` 分配的 sequence 顺序与设备实际观察到的提交顺序不一致，导致设备拒绝 `INVALID_ARGUMENT`。

**修复**：新增 `g_submit_mutex`（独立于 `g_mutex`，避免锁序死锁），在 `submit_command()` 内用临界区包裹 sequence 分配和 `aecDeviceSubmit` 调用。设备始终观察到严格递增的 sequence。

### 6.3 Stream/Event 句柄地址复用（ABA）防御

**问题**：`aecStreamDestroy`/`aecEventDestroy` 从 live registry 移除后立即 `delete`，释放的地址可能被 `new` 重用。旧句柄的指针值可对应新 live 句柄，违反"destroy 后永远 INVALID_HANDLE"契约。

**修复**：添加静态退役容器 `g_retired_streams`、`g_retired_events`（`vector<unique_ptr<T>>`），destroy 时将句柄所有权转入退役容器。对象继续存活直至进程退出，释放的地址永不与新创建的 live 句柄碰撞。生命周期有界——OS 在进程退出时回收内存。所有检索 API 仍独立查 live registry。

### 6.4 未修改

- 同步 eager 模型不变
- 测试断言不变
- grader/public/image 不变
